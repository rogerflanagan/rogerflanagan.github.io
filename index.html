GETTING STARTED

Markup and Style

The main purpose of Eros is to enable designers and developers to create a uniform look and feel across all Zoosk applications.

Namespacing

In order to accomplish this goal, we’ve chosen to use very specific naming techniques. These allow us to keep our code base flat, with low specificity, and keeps us from fighting specificity wars that start with frustration and end with !important. Though we base our naming on the BEM method, we have a few additions of our own outlined below.

BEM

Tokens

Design tokens are the visual design atoms of Eros — specifically, they are named entities that store visual design attributes. We use them in place of hard-coded values (such as hex values for color or pixel values for spacing) in order to maintain a scalable and consistent visual system for UI development.

Mixins

A mixin, unlike a class, represents a behavior. This is a group of styles that represents how objects work, not what they are. A mixin usually doesn't exist by itself; instead, it is included in classes that want to behave in a certain fashion. We have a few @extend's peppered throughout Zephyr. These are in the process of being removed. Please do not use extends at all!

Components

Eros reflects the patterns and components that underpin all Zoosk applications. These patterns and components provide a unified language and consistent look and feel when designing apps and products within the Zoosk ecosystem.

Component States

When a component has a variety of states, we add a class to indicate the state the component, or certain portions of the component, are in. Some examples are: .is-selected, .is-active, .is-expanded, .is-nested, .is-open, .has-focus, .has-error, etc.

Nesting

We want to keep an extremely flat css specificity across the site. Check out this article about the Inception Rule. There will be times when you will nest a class (specified below) and other than that you should not be nesting.

  1. Pseudo Elements & States (::before, :hover)
  2. Modifier Classes (is-*) (has-*)

Refactoring

We have a lot of css here at Zoosk. A lot. During this transition we will be doing a good amount of refactoring to decouple common patterns so that they can be used site wide.

Legacy Code

Instead of retrofitting all of our legacy code, we are going to identify a pattern, and make a brand new style structure, and add it to the trove directory. That will now be the new source of truth moving forward. Then we can look back at the legacy code, identify html/css that looks the same visually, but built differently, and refactor it using the new pattern.

Bug Fixes

When a bug ticket is assigned to you, take the opportunity to look around and not only fix the bug, but see if we can refactor any of the html/css to use a trove pattern. Small cleanup items like this will go a long way.
